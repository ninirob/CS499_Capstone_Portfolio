Live — Milestone Submitted

A CIFAR-10 image classification CNN, originally built for CS 370, enhanced with batch normalization and ReduceLROnPlateau learning rate scheduling, and evaluated with a controlled before-and-after convergence comparison.

Before & after

Before enhancement

The original notebook carried two bugs that never surfaced as errors, only as quietly wrong results. A convolutional layer was written as Conv2D(64, 3, 3), which Keras parses as a kernel size of 3 with a stride of 3, rather than the intended (3,3) kernel with default stride. That silently broke the layer's downsampling and dropped the padding='same' every sibling layer used. Separately, the code meant to report the augmented model's test results had its model.evaluate() call commented out, so the printed accuracy and loss were actually leftover values from an earlier baseline model, not the model that had just finished training.

After enhancement

Both bugs are fixed. The deeper network is now built through a single reusable function that can be instantiated with or without batch normalization, which made it possible to isolate exactly one variable, batch normalization plus a ReduceLROnPlateau learning rate schedule, and run a controlled before-and-after comparison on identical data and architecture. A third bug turned up along the way: a single augmented-data generator was being reused across two separate training runs, and Keras's newer data-handling layer doesn't reliably treat that generator as an infinite stream across repeated fit() calls the way older versions did, so some epochs were silently training on only a handful of batches. That's fixed by wrapping the generator in a tf.data.Dataset with .repeat() and giving each model its own independent dataset instance.

What the results actually showed

Once training was running correctly, the enhanced model scored 66.0% test accuracy against the unenhanced model's 69.2%, with a higher test loss as well, 1.010 versus 0.908. Rather than treat that as a failure to write around, the honest read is in the training logs: batch normalization in this architecture sits directly in the same block as dropout, a combination with documented research showing it can destabilize training, and the learning rate schedule didn't trigger its first drop until epoch 19 of 20, leaving almost no time to benefit from it before training ended. Neither model had fully converged by epoch 20, so this comparison is a snapshot mid-training rather than a final verdict on the approach. The result is a genuine finding, not a bug, and it's a reminder that a well-regarded technique still has to be evaluated in context rather than assumed to help.
