Software Design & Engineering

Live — Milestone Submitted

The Pet BAG Java application, originally built for IT 145, refactored from a flattened single-class design into an abstract class hierarchy with Dog and Cat subclasses and a new KennelManager class.

Try it live

This is the actual enhanced code, running in a real Java compiler. Click Run to compile and see the check-in output for both a Dog and a Cat.

What changed

Structure

Abstract class hierarchy restored

Pet.java went from a flattened design using a petType string field back to a true abstract superclass, with Dog and Cat as concrete subclasses.

New type

Cat added as a first-class type

The original system only ever implemented a Dog path in practice. Cat now exists with its own flat-rate boarding fee logic, not a workaround.

Design

Polymorphism over string checks

KennelManager assigns spaces using instanceof and method overrides instead of comparing petType strings, and calculateBoardingFee() is abstract, so each subclass owns its own pricing logic.

Encapsulation

Space tracking moved out of Pet

The static class-level counters that used to live inside Pet now belong to a dedicated KennelManager class, removing shared mutable state from the Pet hierarchy.

Validation

Input validation added

Constructors now reject empty names, non-positive ages or stay lengths, and invalid breed or weight values, instead of silently accepting bad data.

Bug fix

Constructor mismatch resolved

Pet.java and Dog.java originally came from two separate assignments that were never integrated, causing a compile-breaking constructor mismatch. That's fixed, and a missing printInfo() method was added.
